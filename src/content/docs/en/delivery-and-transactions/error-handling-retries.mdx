## The idea in one sentence

Robust Kafka code sorts every failure into one of three buckets — **retriable** (let the producer retry within `delivery.timeout.ms`), **abortable** (roll the transaction back and try again), or **fatal** (close the producer and let another instance take over) — and dead-letters the records that can never succeed.

## Producer retries are automatic and time-bounded

The producer retries retriable errors — leader elections, transient network blips — on its own. The modern control is not a retry *count* but a **time budget**: `delivery.timeout.ms` (default **120000**) is the upper bound on how long `send()` may take to report success or failure, covering batching, broker acknowledgement, and every retry underneath. It must be at least `request.timeout.ms + linger.ms`.

```properties
delivery.timeout.ms=120000     # total deadline for a send, including all retries
request.timeout.ms=30000       # per-attempt broker wait
retries=2147483647             # effectively unlimited; the deadline bounds it
enable.idempotence=true        # retries stay duplicate-free within a partition
```

With `enable.idempotence=true`, these retries never create duplicates in the partition, so you can retry aggressively without corrupting the stream.

## Retriable vs abortable vs fatal in a transaction

Inside a transactional loop, exceptions are not all equal — the recovery action differs by class:

- **`TransactionAbortableException`** — the transaction cannot commit but the producer is still healthy. Call `abortTransaction()` and **retry the batch** from the last committed offsets.
- **`ProducerFencedException`** and **`OutOfOrderSequenceException`** — **fatal**. Another instance has claimed your `transactional.id` (a zombie was fenced) or the idempotent sequence is broken. You cannot recover this producer; **close it and shut down** so a healthy instance continues.

```java
try {
    producer.beginTransaction();
    for (ConsumerRecord<String, String> record : records) {
        producer.send(new ProducerRecord<>("orders-enriched", record.key(), transform(record)));
    }
    producer.sendOffsetsToTransaction(offsetsFor(records), consumer.groupMetadata());
    producer.commitTransaction();
} catch (ProducerFencedException | OutOfOrderSequenceException e) {
    producer.close();       // FATAL: a zombie was fenced — give up this instance
    shutdown();
} catch (TransactionAbortableException e) {
    producer.abortTransaction();   // roll back and retry the batch
}
```

## Poison messages, dead-letter topics, and the safety net

A **poison message** is a record the consumer can never process — malformed payload, an unparseable schema, a permanently failing downstream. Retrying it forever stalls the whole partition behind it. The standard fix is a **dead-letter topic (DLT)**: after N failed attempts, produce the record plus error context to a separate `orders-dlq` topic, commit past it, and keep the main flow moving. A human or a separate job inspects the DLT out of band.

```java
try {
    process(record);
} catch (NonRetriableException e) {
    // Route the bad record aside instead of blocking the partition
    dlqProducer.send(new ProducerRecord<>("orders-dlq", record.key(), record.value()));
}
// commit past the record either way, so the partition keeps advancing
```

Underneath all of this, **idempotent consumers are the safety net**. Because at-least-once means a record can be redelivered after any crash, make your processing safe to repeat — an `UPSERT` keyed by an event id, a conditional write, a dedupe table. Then a retry, an aborted-and-retried batch, or a redelivery after a fenced producer all converge on the same correct state.
