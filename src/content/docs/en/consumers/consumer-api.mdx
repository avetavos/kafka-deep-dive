## The idea in one sentence

A Kafka consumer subscribes to topics and then **loops on `poll()`** — each poll returns a batch of records from the partitions assigned to it, and that same poll call is what keeps the consumer alive in its group.

## Subscribe, then poll

You configure a consumer with `bootstrap.servers`, a `group.id`, and **deserializers** (the mirror of the producer's serializers). Then you `subscribe()` to one or more topics and drive a **poll loop**:

```java
Properties props = new Properties();
props.put("bootstrap.servers", "localhost:9092");
props.put("group.id", "order-processor");          // consumer group membership
props.put("key.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");
props.put("value.deserializer", "org.apache.kafka.common.serialization.StringDeserializer");

try (KafkaConsumer<String, String> consumer = new KafkaConsumer<>(props)) {
    consumer.subscribe(List.of("orders"));
    while (true) {
        // Wait up to 1s for records, then process whatever arrived
        ConsumerRecords<String, String> records = consumer.poll(Duration.ofMillis(1000));
        for (ConsumerRecord<String, String> record : records) {
            System.out.printf("p=%d off=%d key=%s val=%s%n",
                record.partition(), record.offset(), record.key(), record.value());
        }
    }
}
```

## Why the loop matters

`poll()` does far more than fetch data. On each call the consumer also sends **heartbeats**, participates in group coordination, and fetches its partition assignments. If you stop polling — say, because processing a batch takes too long — the group coordinator assumes the consumer is dead and triggers a **rebalance**, reassigning its partitions elsewhere. The rule: **keep polling, and keep per-batch processing bounded.**

## Each record carries its coordinates

Every `ConsumerRecord` exposes its `topic()`, `partition()`, `offset()`, `key()`, `value()`, `timestamp()`, and headers. Those coordinates are what you use to commit progress and to reason about ordering — the consumer sees records **in offset order within each partition**, though records from different partitions may interleave across poll batches.

<Callout title="The one-line takeaway">
A consumer **`subscribe()`s** to topics and drives a **`poll()` loop**. `poll()` returns a batch **and** does the invisible work — heartbeats, group coordination, fetching assignments — so if you stop polling or block too long, the group **rebalances** your partitions away. Records arrive in **offset order within a partition**, each tagged with topic, partition, and offset.
</Callout>
