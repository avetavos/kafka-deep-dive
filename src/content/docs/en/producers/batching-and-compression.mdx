## The idea in one sentence

The producer **groups records into batches** and optionally **compresses** them, so a little extra waiting (`linger.ms`) and bigger batches (`batch.size`) buy dramatically higher **throughput** — as long as writes stay inside `delivery.timeout.ms`.

## linger.ms and batch.size trade latency for throughput

The producer does not send each record on its own. It accumulates records per partition into a **batch** and sends the batch as one request. Two settings shape that batch:

- **`batch.size`** — the maximum size in **bytes** of a single batch. A batch ships when it fills up to this size (or when `linger.ms` elapses).
- **`linger.ms`** — how long the producer will **wait** for more records before sending a not-yet-full batch. It defaults to `0` (send as soon as possible). Raising it to, say, `10` lets more records accumulate, producing fuller batches.

Larger batches mean fewer requests, better compression, and higher throughput — at the cost of a few extra milliseconds of latency per record. That is the central lever.

```properties
# Wait up to 10 ms to fill a batch, and allow bigger batches for throughput
linger.ms=10
batch.size=65536
```

## compression.type shrinks batches on the wire

Compression is applied **per batch**, so bigger batches compress better. The `compression.type` config picks the codec, each a different point on the speed-versus-ratio curve:

- **`lz4`** — very fast, good ratio; a common default choice.
- **`zstd`** — excellent ratio, still fast; great when network or storage is the bottleneck.
- **`snappy`** — fast, modest ratio.
- **`gzip`** — best ratio, but the most CPU-heavy and slowest.
- **`none`** — no compression (the default).

```properties
# Compress each batch; lz4 and zstd are the usual picks for throughput
compression.type=lz4
```

## buffer.memory, back-pressure, and the delivery deadline

Because `send()` is asynchronous, records sit in an in-memory buffer before the I/O thread transmits them. `buffer.memory` caps the total size of that buffer. When it fills — because the app produces faster than the network can drain — `send()` **blocks** (up to `max.block.ms`), applying **back-pressure** so a fast producer cannot exhaust memory.

Two timeouts bound the whole journey. `delivery.timeout.ms` (default **120000**) is the **total deadline** from the moment `send()` returns until the record is acknowledged — it covers batching delay, network sends, **and all retries**. It should be **>= `request.timeout.ms` + `linger.ms`**. If a record cannot be delivered within it, the send fails permanently.

```properties
# 32 MiB send buffer; when full, send() applies back-pressure and blocks
buffer.memory=33554432
# Total deadline covering batching + retries; must be >= request.timeout.ms + linger.ms
delivery.timeout.ms=120000
request.timeout.ms=30000
```

## Tuning for throughput

To push throughput up, raise `linger.ms` a little (5-20 ms), enlarge `batch.size`, and turn on `lz4` or `zstd` compression — the three work together, since more waiting makes fuller batches and fuller batches compress better. Keep `buffer.memory` comfortable so back-pressure does not stall you, and make sure `delivery.timeout.ms` is generous enough to absorb retries during a broker hiccup.

<Callout title="The one-line takeaway">
The producer batches records: **`linger.ms`** and **`batch.size`** trade a little **latency** for much higher **throughput**, and **`compression.type`** (**lz4**, **zstd**, snappy, gzip) shrinks each batch. **`buffer.memory`** applies **back-pressure** when full, and **`delivery.timeout.ms`** (default **120000**) is the total send deadline covering all retries.
</Callout>
