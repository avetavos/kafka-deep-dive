## ไอเดียในหนึ่งประโยค

guarantee แบบ end-to-end แข็งแรงได้เท่ากับจุดที่อ่อนที่สุดของมันเท่านั้น ฝั่ง **producer** (`acks` และ idempotence) กับฝั่ง **consumer** (จังหวะ commit) ต่างก็มีส่วน และคุณต้องคิดถึงมัน **พร้อมกัน**

## guarantee เดียวที่มีสองครึ่ง

delivery semantics ไม่ใช่สวิตช์ตัวเดียว data ไหลจาก producer → log → consumer และแต่ละ hop มี failure mode ของตัวเอง

- **Producer → log** ด้วย `acks=all` และ `enable.idempotence=true` record ถูกเขียนแบบทนทานและ retry ไม่สร้าง duplicate ใน partition ถ้าใช้ `acks=0` หรือ `acks=1` broker ล่มจังหวะไม่ดีก็ทำให้ write หายได้
- **Log → consumer** commit **หลัง** ประมวลผลได้ at-least-once commit **ก่อน** ได้ at-most-once ตัว log เองไม่เคยทำ record ที่ commit แล้วหายภายใน retention

รวมสองอย่างเข้าด้วยกันก็ได้พฤติกรรม end-to-end จริง ๆ

| Producer | Consumer commit | End-to-end |
| --- | --- | --- |
| `acks=all` + idempotent | หลังประมวลผล | at-least-once (ไม่หาย อาจซ้ำ) |
| `acks=1` | หลังประมวลผล | at-least-once แต่ broker ล่มแบบหายากยังทำ write หายได้ |
| `acks=all` + idempotent | ก่อนประมวลผล | at-most-once (ไม่ซ้ำ อาจหาย) |
| transactional producer | commit offset ใน transaction | exactly-once ภายใน Kafka |

## default ที่ควรออกแบบเผื่อไว้

ระบบส่วนใหญ่รันแบบ **at-least-once end-to-end** คือ `acks=all` producer เป็น idempotent commit หลังประมวลผล แล้วทำให้ผลลัพธ์ปลายทาง **idempotent** เพื่อให้ duplicate ไม่เป็นอันตราย มันเรียบง่าย ทนทาน และเร็ว หยิบ **exactly-once** เต็มรูปแบบ (transactions) มาใช้ก็ต่อเมื่อ pipeline อยู่ภายใน Kafka และ duplicate ยอมรับไม่ได้จริง ๆ เพราะมันเพิ่มต้นทุนการ coordinate ส่วนที่เหลือของ module นี้จะค่อย ๆ สร้างเส้นทาง exactly-once นั้นขึ้นมา

<Callout title="สรุปหนึ่งบรรทัด">
guarantee แบบ end-to-end คือ **การรวมกัน** ของครึ่ง producer (`acks=all` + idempotence เพื่อ write ที่ไม่หายไม่ซ้ำ) กับครึ่ง consumer (จังหวะ commit สำหรับ at-most vs at-least-once) default ที่ทนทานคือ **at-least-once end-to-end บวกกับ sink ที่ idempotent** ส่วน **exactly-once** จริง ๆ มาจาก Kafka **transactions** และคุ้มค่ากับต้นทุนก็ต่อเมื่อ duplicate ยอมรับไม่ได้
</Callout>
