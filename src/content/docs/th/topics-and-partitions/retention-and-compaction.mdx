## ไอเดียในประโยคเดียว

Kafka ไม่ได้เก็บข้อมูลตลอดไปโดย default — **cleanup policy** ของ topic ตัดสินว่า record เก่าจะถูก **ลบ** หลังพ้น retention window หรือถูก **compact** ให้เหลือแค่ **ค่าล่าสุดต่อ key**.

## Retention: ลบตามเวลาหรือขนาด

ด้วย policy default `cleanup.policy=delete` Kafka ลบ log segment เก่าเมื่อเกิน threshold:

```properties
# Keep records for 7 days (whichever limit hits first)
cleanup.policy=delete
retention.ms=604800000
# ...or cap the partition at 50 GB
retention.bytes=53687456000
```

Retention บังคับที่ระดับ **segment** ทั้งก้อน ไม่ใช่ทีละ record และดูที่ **อายุหรือขนาด** — ไม่ได้ดูว่ามีใคร consume ไปแล้วหรือยัง consumer ที่ตามหลังเกิน retention window จะพลาด record ที่หมดอายุไป policy นี้เหมาะกับ event stream ที่ event เก่าหมดคุณค่า: clickstream, log, metric.

## Compaction: เก็บค่าล่าสุดต่อ key

ตั้ง `cleanup.policy=compact` แล้ว Kafka เปลี่ยนกลยุทธ์ทั้งหมด แทนที่จะทิ้งข้อมูลเก่าตามอายุ มันเก็บ **record ล่าสุดของแต่ละ key** และค่อยๆ garbage-collect ค่าเก่าของ key นั้น:

```text
before compaction:  (a,1)(b,1)(a,2)(c,1)(b,2)(a,3)
after compaction:   (c,1)(b,2)(a,3)     ← latest value per key survives
```

แบบนี้เปลี่ยน topic ให้เป็น **snapshot ของ state ปัจจุบันตาม key** ที่ durable และ replay ได้ — เหมาะกับ changelog, config และ state store ที่ Kafka Streams เก็บ consumer ใหม่เอี่ยมอ่านทั้ง compacted topic แล้วสร้าง state ล่าสุดของทุก key ขึ้นมาใหม่ได้.

## Tombstone: ลบ key

ภายใต้ compaction จะลบ key ทั้งอันยังไง? เขียน **tombstone** — record ที่มี key และ **value เป็น null** compaction มองว่า "key นี้หายไปแล้ว" เก็บมันไว้นานพอให้ consumer ทุกตัวเห็นการลบ แล้วค่อยลบทั้ง tombstone และ history ของ key นั้น.

```typescript
// Tombstone: null value tells a compacted topic to forget this key
await producer.send({
  topic: 'user-config',
  messages: [{ key: 'user-42', value: null }],
})
```

รวม policy ได้ด้วย `cleanup.policy=compact,delete` เพื่อ compact ตาม key **และ** ยังทยอยลบข้อมูลที่เก่ามากออกไป.

## เลือก policy

- **`delete`** — ตัว record คือประเด็น history สำคัญ ค่าล่าสุดต่อ key ไม่สำคัญ เหมาะกับ event stream.
- **`compact`** — **ค่าล่าสุดของ key** คือประเด็น อยากได้ snapshot ที่สร้างใหม่ได้ เหมาะกับ state, changelog, config topic.

<Callout title="สรุปหนึ่งบรรทัด">
**`cleanup.policy`** ของ topic เลือก lifecycle ของมัน **`delete`** ทยอยลบ record ตาม **`retention.ms`** / **`retention.bytes`** โดยไม่สนการ consume — เหมาะกับ event stream **`compact`** เก็บ **ค่าล่าสุดต่อ key** เปลี่ยน topic เป็น state snapshot ที่สร้างใหม่ได้ — เหมาะกับ changelog และ config ส่วน **tombstone** (value null) ลบ key ภายใต้ compaction.
</Callout>
