## แนวคิดในหนึ่งประโยค

**consumer group** คือกลุ่มของ consumer ที่แชร์ partition ของ topic กัน — แต่ละ partition ถูกจัดการโดย consumer เพียงตัวเดียวใน group — และ **rebalancing** คือกระบวนการจัดสรร partition ใหม่เมื่อสมาชิกใน group เปลี่ยนไป.

## group แบ่งงานกันทำ

consumer ทุกตัวประกาศ `group.id` Kafka จะยก **partition** แต่ละอันให้ consumer เพียง **ตัวเดียว** ภายใน group ดังนั้น topic ที่มี 6 partition จะถูกประมวลผลพร้อมกันได้สูงสุด 6 consumer เพิ่ม consumer ตัวที่ 7 เข้าไปมันจะนั่งว่าง — เพดานของ parallelism คือจำนวน partition ไม่ใช่จำนวน consumer.

group ต่างกันเป็นอิสระต่อกัน: `analytics` กับ `billing` ต่างได้ stream เต็ม ๆ ของตัวเอง และ track offset ของตัวเอง ส่วนภายใน group เดียวกัน partition จะถูกแบ่งกัน.

คุณตรวจดู group, สมาชิก และ lag ได้จาก CLI:

```bash
# Show members, assigned partitions, and lag for a group
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --describe --group billing
```

## Rebalancing

เมื่อ consumer เข้า, ออก หรือตาย — หรือมีการเพิ่ม partition — group ต้อง **rebalance**: กระจาย partition ใหม่ไปยังสมาชิกที่มีอยู่ตอนนั้น protocol แบบ classic ทำสิ่งนี้ด้วยขั้นตอน **stop-the-world**: consumer ทุกตัวหยุดประมวลผล คืน partition ทั้งหมด แล้วรอ assignment ใหม่ บน group ขนาดใหญ่ การหยุดแบบนั้นเจ็บมาก.

## KIP-848: rebalance protocol ตัวใหม่

Kafka 4.0 มาพร้อม **next-generation consumer rebalance protocol (KIP-848)** ในสถานะ **generally available** มันย้าย logic การ assign ไปไว้ที่ **group coordinator** ฝั่ง broker และทำให้ rebalance เป็นแบบ **incremental** — เพิกถอนเฉพาะ partition ที่ต้องย้ายจริง ๆ ส่วนที่เหลือประมวลผลต่อได้ ผลลัพธ์คือ rebalance สั้นลงและรบกวนงานน้อยลง โดยเฉพาะกับ group ขนาดใหญ่.

```properties
# Opt into the new consumer group protocol (KIP-848), GA in Kafka 4.0
group.protocol=consumer
# The classic (older) protocol is still available:
# group.protocol=classic
```

<Callout title="ถ้าคุณเรียน Kafka เมื่อหลายปีก่อน">
คำแนะนำเดิมคือ "เลี่ยง rebalance เพราะมัน stop the world" แต่ด้วย KIP-848 (`group.protocol=consumer`, GA ใน Kafka 4.0) rebalance เป็นแบบ **incremental และขับโดย broker** — partition ที่ไม่โดนแตะยังไหลต่อเนื่อง เมื่อเปิดใช้แล้ว group จะ downgrade ได้ถึงแค่ Kafka 3.4.1 หรือใหม่กว่าเท่านั้น.
</Callout>

<Callout title="สรุปหนึ่งบรรทัด">
**consumer group** แบ่ง partition ของ topic ให้สมาชิก — หนึ่ง partition ต่อหนึ่ง consumer ดังนั้นจำนวน partition จึงเป็นเพดานของ parallelism — ส่วน group ที่แยกกันต่างอ่าน stream ทั้งเส้น การเปลี่ยนสมาชิกจะ trigger **rebalance** และ Kafka 4.0 ทำให้ราบรื่นขึ้นมากด้วย protocol **KIP-848** (`group.protocol=consumer`) ที่ขับโดย broker และเป็น **incremental** แทน stop-the-world.
</Callout>
